# Steering Schema

**Consumed by**: `onboard-project`, `write-spec`, `verify-code`, `open-pr`.

Steering documents live at the project root under `steering/` and carry project-specific context that skills must not hardcode. Every steering doc is plain Markdown; skills read them via the `Read` tool and treat them as the abstraction layer between stack-agnostic workflow logic and project specifics.

## Roster

| Document | Purpose | Read by |
|----------|---------|---------|
| `steering/product.md` | Vision, mission, personas, feature prioritization (MoSCoW), user journeys | `draft-issue` (Step 1 context), `write-spec` Phase 1 (user context), `onboard-project` (creates) |
| `steering/tech.md` | Stack, performance/security constraints, testing standards, verification gates, versioning matrix, coding conventions, environment variables | `write-spec` Phase 2 (architecture), `write-code` (implementation rules), `verify-code` (verification gates), `open-pr` Step 2 (bump matrix), `draft-issue` Step 4 (codebase investigation) |
| `steering/structure.md` | Project layout, layer architecture, naming conventions, architectural invariants, import order, test project scaffolding | `write-spec` Phase 2 (design patterns), `write-code` (file-path conventions), `verify-code` (architectural invariants) |
| `steering/retrospective.md` | Defect-derived learnings applicable to future spec writing (generated by `$nmg-sdlc:run-retro`) | `write-spec` Phase 1 (applies learnings to new ACs) |

## Discovery

Consuming skills file discovery for `steering/*.md` before reading. A missing steering doc is acceptable in most paths — consumers degrade gracefully (e.g., `write-spec` omits retrospective-driven AC amendments when `retrospective.md` is absent). The one hard requirement: `$nmg-sdlc:write-spec` expects `product.md`, `tech.md`, and `structure.md` to exist. If any is missing, suggest `$nmg-sdlc:onboard-project` and do not attempt to substitute.

## Read timing

- **Product context** is read at the start of interview/discovery workflows (`draft-issue` Step 1, `write-spec` Phase 1) — before any AC drafting.
- **Tech context** is read when touching architecture (`write-spec` Phase 2), implementation (`write-code` Step 3), or release (`open-pr` Step 2).
- **Structure context** is read alongside `tech.md` wherever architectural decisions or file-path conventions apply.
- **Retrospective learnings** are read by `write-spec` Phase 1 and applied as transferable principles — the skill maps each learning's abstract pattern to concrete scenarios for the current feature rather than copying the learning verbatim.

## Invariants

- Steering docs are authoritative for project specifics. Skills that embed project-specific defaults (language, framework, file layout) violate the `steering/structure.md` → "Skills must reference steering docs for project context" invariant.
- Steering docs are human-authored (via `$nmg-sdlc:onboard-project`). Skills may read them but must never rewrite them automatically — `$nmg-sdlc:upgrade-project` is the single mechanism that updates steering docs, and only by adding missing template sections.
- `retrospective.md` is the only steering doc produced by an automated pipeline (`$nmg-sdlc:run-retro`).
