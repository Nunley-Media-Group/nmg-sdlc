## How to Use This Document

This document is automatically generated by `$nmg-sdlc:run-retro` and read by
`$nmg-sdlc:write-spec` during Phase 1 (SPECIFY). When writing new feature specs, apply
these transferable patterns to new feature contexts - adapt each learning to the
current feature's domain rather than applying it literally.

## Missing Acceptance Criteria

Defects caused by scenarios that the original feature spec did not cover at all.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When specifying features reachable by both interactive users and automated orchestrators, specs often cover only the happy path or describe non-interactive behavior in prose. Error branches, confirmation branches, stale-branch branches, and missing-prerequisite branches then inherit blocking prompts or ambiguous defaults under automation. | Include mode-specific ACs for every branch that can pause, confirm, skip, escalate, or mutate state. For automated mode, require deterministic behavior, runner-compatible escalation text, and no interactive prompt calls. | specs/bug-add-auto-mode-support-to-migrate-project-skill/, specs/bug-fix-migrate-project-auto-mode/, specs/bug-replace-askuserquestion-with-escalation-when-specs-missing/, specs/bug-prevent-start-issue-prompts-during-unattended-run-loop-exec/ |
| When specifying features that consume project context, specs often say the feature "reads project context" without naming each document type and where its constraints affect output. The implementation then reads only the most obvious document and misses technical or structural constraints. | List each context/reference source individually, bind each source to a workflow step, and add ACs proving constraints from each source are surfaced in downstream decisions or generated artifacts. | specs/bug-fix-draft-issue-steering-docs/ |
| When specifying features that reconcile user artifacts against evolving templates or schemas, specs often treat "missing section" as sufficient reason to add content. This omits contextual relevance, per-item approval, and durable user exclusions. | Require relevance inference from codebase evidence, granular approval for ambiguous additions, and persistent exclusions that distinguish intentionally declined content from genuinely new template content. | specs/bug-fix-migrate-project-irrelevant-sections/ |
| When specifying features that create or tolerate runtime files in a user-controlled workspace, specs often define the artifact's purpose but omit ACs for version-control exclusion and sibling-skill dirty-tree filtering. Runtime files then look like user changes or get committed. | Define the runtime artifact inventory, require idempotent VCS exclusion for generated local state, and require dirty-tree checks to filter known runtime artifacts while still rejecting real user changes. | specs/bug-fix-sdlc-runner-auto-mode-gitignore/, specs/bug-fix-start-issue-dirty-tree-runner-artifacts/, specs/bug-fix-open-pr-to-abort-on-dirty-tree-or-empty-implementation-branch/ |
| When specifying features that select work from an external collection, specs often cover a normal candidate but omit empty queues, unsupported metadata fields, and pre-filtering rules. Selection then fails closed, falls through, or treats every candidate as blocked. | Specify the supported metadata contract, unsupported-field behavior, empty-queue terminal behavior, pre-filtering rules, and fallback diagnostics before any child workflow is spawned. | specs/bug-fix-start-issue-milestone-selection/, specs/bug-run-loop-no-automatable-issue-fallthrough/, specs/bug-drop-unsupported-parent-field-from-gh-issue-view-json-query-in-sdlc-runner/ |
| When specifying features that read optional upstream artifacts, specs assume the full workflow ran and omit the "artifact absent" branch. Legitimate partial workflows then hard-fail instead of using an available fallback. | Add ACs for detecting missing artifacts before reading, naming fallback data sources, adapting output when fallback is used, and preserving the richer path when artifacts are present. | specs/bug-handle-missing-spec-files-in-open-pr/ |
| When specifying orchestrators that evaluate child-step success, specs often define success through one signal type only. Exit codes, JSON result events, textual failure markers, current tool output, and quoted historical context are not separated, so silent failures and false positives both cascade. | Define the full failure-detection surface in ACs: structured result fields, text patterns, contextual false-positive carve-outs, and postcondition checks. Require current live failures to be distinguished from quoted memory, specs, logs, or assistant prose. | specs/bug-detect-soft-failures-runner-tests/, specs/bug-text-pattern-soft-failure-detection-sdlc-runner/, specs/bug-fix-false-positive-permission-denied-escalation/, specs/bug-prevent-github-soft-failure-matching-from-retrying-on-quoted-historical-text/, specs/bug-fix-run-loop-child-codex-github-access/, specs/bug-fix-unattended-sdlc-runner-cascade-failure-on-diverged-remote-branch/ |
| When specifying features that mutate shared workspace state, specs focus on the intended mutation but omit preconditions for ambient state. Dirty worktrees, empty implementation branches, stale remote branches, and diverged histories then get rewritten or pushed without a deliberate recovery contract. | Specify readiness predicates before mutation, fail-fast diagnostics, branch/history recovery rules, and the exact safe push behavior for rewritten history. Require these checks to happen before version bumps, checkouts, pulls, cleanup, or PR creation. | specs/bug-validate-git-cleanliness-before-branching-in-start-issue/, specs/bug-run-loop-start-cycle-destructive-cleanup/, specs/bug-fix-open-pr-to-abort-on-dirty-tree-or-empty-implementation-branch/, specs/bug-fix-unattended-sdlc-runner-cascade-failure-on-diverged-remote-branch/ |
| When specifying continuous orchestrators, specs often cover successful loop continuation but omit the terminal condition for a single work item escalating. The runner can preserve global loop behavior while losing the failed issue as the manual recovery target. | Add ACs that escalation from any step stops the current invocation, preserves the failed issue and branch as the recovery target, and only continues to the next issue after a fully successful cycle. | specs/bug-fix-run-loop-continuing-after-issue-escalation/ |
| When specifying spec-writing from issue bodies, specs often assume the issue is already complete enough to formalize. Open questions, malformed ACs, or defect reports without reproduction details then become underspecified requirements instead of triggering a bounded interview. | Add gap-detection ACs before requirements drafting: unresolved open questions, malformed or missing ACs, and defect-specific gaps. In interactive mode, ask bounded targeted questions; in unattended mode, log the bypass and carry unresolved gaps forward visibly. | specs/bug-fix-write-spec-to-interview-users-when-the-issue-has-open-questions/ |
| When specifying features that discover specs or traceability links, specs often define that a field exists but omit the active lookup procedure and strict match rules. Implementations then leave the field empty or pick an unrelated newest directory. | Specify the discovery surfaces, matching precedence, canonical issue/branch/frontmatter checks, null behavior, and a hard rejection path for unrelated candidates. | specs/bug-fix-write-spec-defect-related-spec-search/, specs/bug-runner-spec-validation-accepts-unrelated-specs/ |
| When specifying user-facing configuration or documentation, specs can accidentally document controls the runtime does not implement. Users copy a contract that looks supported but is ignored. | Include ACs that example configs and docs are validated against implemented runtime behavior, and add regression checks preventing unsupported fields or options from being reintroduced. | specs/bug-unsupported-maxturns-runner-config-contract/ |

## Undertested Boundaries

Defects caused by boundary conditions the original feature spec addressed
insufficiently.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When specifying features that delegate work to subprocesses, tools, or LLM sessions, specs often trust the delegate's self-reported success. Delegates can exit cleanly after partial work, quota exhaustion, omitted commits, or failed pushes. | Pair every delegate success signal with independent postcondition verification: expected files, branch, commits, version changes, PR state, and structured result fields. Retry or escalate when the postcondition, not just the delegate's report, fails. | specs/bug-detect-soft-failures-runner-tests/, specs/bug-fix-silent-commitpush-failure/, specs/bug-fix-inconsistent-version-bumping/ |
| When specifying transient state lifecycle, specs cover normal start and normal completion but omit signals, escalations, crashes, single-step exits, and descendant subprocesses. State and child processes then leak across sessions. | Enumerate every termination route and require centralized best-effort cleanup for files and process trees, scoped only to artifacts and descendants the feature created. | specs/bug-fix-auto-mode-cleanup-on-exit/, specs/bug-fix-sdlc-runner-process-tree-cleanup/ |
| When specifying phases that monitor asynchronous external status, specs often cover pass and fail but underdefine pending, absent, and "reported but unresolved" states. The phase can exit successfully after observation or proceed before a terminal pass. | Distinguish present-passing, present-failing, pending, and absent signals. Tie success to the desired terminal state, allocate enough budget for remediation, and define skip or escalation behavior for unavailable integrations. | specs/bug-fix-monitorci-step8/, specs/bug-fix-sdlc-runner-infinite-retry-no-ci/, specs/bug-runner-merge-progression-on-pending-ci/ |
| When specifying loops over multiple work units, specs say state is reset between iterations but omit the canonical source for each next identifier. Free-form output from a prior iteration can contaminate the next issue, spec, or branch selection. | Require identifiers to come from deterministic ground truth such as issue metadata, filesystem state, branch names, or frontmatter. Validate derived identifiers against that source before advancing. | specs/bug-fix-sdlc-runner-cross-cycle-state-contamination/ |
| When specifying cross-reference traversal, specs cover the normal link but not malformed, circular, wrong-type, or chained links. Reference fields then silently point to the wrong class of artifact or stop before the root target. | Add ACs for target existence, expected type, recursive chain resolution, cycle detection, broken-chain warnings, and canonical root replacement. | specs/bug-fix-run-retro-defect-spec-discovery/, specs/bug-fix-write-spec-defect-related-spec-search/ |
| When specifying path-based configuration, specs often cover the current layout but omit installed-cache layout, legacy layout, stale absolute paths, and valid custom roots. The first environment change turns configuration into a startup blocker. | Specify root-shape validation, precedence when multiple path fields exist, backward compatibility for legacy fields, stale-path refresh or actionable diagnostics, and preservation of valid custom roots. | specs/bug-fix-sdlc-runner-path-resolution-to-support-the-cc-plugin-cache-layout/, specs/bug-fix-stale-pluginroot-config-blocking-sdlc-loop/ |
| When specifying branch and remote-history recovery, specs cover fresh branches but omit stale remotes, concurrent base advancement, rebase rewrites, and safe force-push conditions. Recovery loops then bounce between steps without a shared reason. | Define stale branch detection at cycle start, structured bounce reasons between steps, division of responsibility for rebase and PR creation, and automatic force-with-lease only when the lease is safe in unattended mode. | specs/bug-fix-unattended-sdlc-runner-cascade-failure-on-diverged-remote-branch/ |
| When specifying dependency ordering, specs often state that issues may block or depend on one another but omit directionality tests for each syntax. Inverse relationship handling makes blocked work appear ready. | Add paired ACs for every supported relationship phrase, explicitly naming which issue becomes blocked and preserving no-regression cases for the inverse phrase. | specs/bug-run-loop-blocks-dependency-ordering/ |
| When specifying queue-driven orchestration, specs often cover blocked and selected work but underdefine the empty-ready-queue terminal case. The runner can continue into a child step with no work selected. | Add ACs for no selected issue, no automatable issues, and blocked-only queues, including whether the runner exits done, logs diagnostics, or escalates. | specs/bug-run-loop-no-automatable-issue-fallthrough/ |
| When specifying merge or delivery gates, specs often validate negative terminal states but omit pending intermediate states. Work can merge while checks are still queued or in progress. | Include every status vocabulary value returned by the external system and classify each as pass, fail, pending-blocker, or absent-skip before merge progression is allowed. | specs/bug-runner-merge-progression-on-pending-ci/ |

## Domain-Specific Gaps

Defects caused by domain knowledge the original feature spec failed to capture.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When a feature composes prompts or instructions for a downstream parser, specs may describe the semantic content but omit positional recognition rules. Tokens that must be first can become prose and never invoke the intended behavior. | Pin exact invocation structure, token position, and where modifiers are appended. Include a regression AC for strict-recognition environments where inference is disabled. | specs/bug-exercise-template-dry-run-prefix/ |
| When specifying features that synthesize guidance from historical artifacts, specs often define inputs and output tables but omit the required abstraction level. The result becomes a catalog of local incidents instead of transferable spec-writing principles. | Specify domain-agnostic framing, cross-source aggregation, evidence traceability, and future-oriented recommendations. Reject rows that merely restate one bug. | specs/bug-fix-run-retro-learnings-transferability/ |
| When specifying artifacts stored in host-controlled filesystem areas, specs may miss platform constraints around reserved directories. A future host protection change then breaks the workflow without a migration path. | Distinguish user-content paths from host-reserved paths, require canonical artifacts live outside protected directories, and define hard gates plus upgrade migration when conventions change. | specs/bug-relocate-specs-and-steering/ |
| When specifying migrations over heterogeneous legacy artifacts, specs often state the target convention but omit detection predicates, history-preserving mechanics, cross-reference rewrites, and destructiveness classifications. Safe changes are skipped while risky ones look equivalent. | Specify each legacy shape, how to classify it, the command semantics required to preserve history, related cross-reference updates, and which transformations can auto-apply versus require human approval. | specs/bug-spec-directory-rename-migrate-project/, specs/bug-add-auto-mode-support-to-migrate-project-skill/, specs/bug-fix-migrate-project-auto-mode/ |
| When specifying nested automation, specs often miss capability inheritance boundaries between parent and child processes. Sandboxes, environment variables, auth, and network access can diverge even when the parent process is healthy. | Define the child capability contract explicitly: which environment variables are stripped, which auth/path/network capabilities must be preserved, which launch mode is required, and how parity is verified. | specs/bug-run-loop-nested-codex-sandbox-inheritance/, specs/bug-fix-run-loop-child-codex-github-access/ |
| When specifying output classifiers, specs often match text without domain context. Benign sandbox denials, quoted historical errors, and literal phrases inside specs can be treated as current failures. | Require classifiers to use structured fields and contextual dimensions such as path, source, current command boundary, and benign exception lists before assigning failure categories. | specs/bug-fix-false-positive-permission-denied-escalation/, specs/bug-fix-verify-step-false-escalation-on-temp-dir-denials/, specs/bug-prevent-github-soft-failure-matching-from-retrying-on-quoted-historical-text/ |
| When specifying integrations with external CLIs or APIs, specs often assume fields and flags exist because they match the conceptual model. Version-specific field availability and unsupported config options then become runtime blockers or misleading docs. | Capture the observed external interface as a contract, include unsupported-field behavior, version compatibility notes, and tests that docs/examples only advertise implemented capabilities. | specs/bug-drop-unsupported-parent-field-from-gh-issue-view-json-query-in-sdlc-runner/, specs/bug-unsupported-maxturns-runner-config-contract/ |
| When specifying release or versioning automation, specs often describe the desired version result but not the deterministic source of truth, commit boundary, or final postcondition. Version bumps become inconsistent across manual and unattended paths. | Define the bump source data, artifact list, commit timing, manual-vs-automated responsibility split, and a final diff/postcondition check before PR creation is considered complete. | specs/bug-fix-inconsistent-version-bumping/ |
