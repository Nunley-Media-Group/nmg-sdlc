## Missing Acceptance Criteria

Defects caused by scenarios that the original feature spec did not cover at all.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When specifying features reachable by both interactive users and automated orchestrators, specs often state non-interactive support as "out of scope" in prose or cover only the happy path under each mode — every branch (success, error, missing prerequisite, destructive subset) then inherits interactive defaults (blocking prompts) and silently hangs or prompts under automation | Include positive ACs that enumerate every execution branch (success, error, missing-prerequisite, destructive vs safe) under each mode; require non-interactive paths to emit a machine-readable escalation sentinel rather than any blocking prompt; assert that cross-cutting mode flags produce deterministic, mode-specific behavior rather than being silently inherited from project-wide conventions | specs/bug-add-auto-mode-support-to-migrate-project-skill/, specs/bug-fix-migrate-project-auto-mode/, specs/bug-replace-askuserquestion-with-escalation-when-specs-missing/ |
| When specifying features that consume multiple reference/context documents of different types, the spec often describes context-gathering in the aggregate ("reads project context") without enumerating each document type and where in the workflow it is consulted — implementation then wires in only the most prominent document | List each context/reference source individually in ACs and bind each to a specific workflow step where it must be consulted; include ACs verifying that constraints drawn from each source materially affect downstream output rather than just confirming the file was read | specs/bug-fix-draft-issue-steering-docs/ |
| When specifying features that reconcile a user's artifact against an evolving template or schema, specs treat "missing vs. present" as the sole relevance signal and omit ACs for filtering proposed additions by contextual evidence, for per-item user approval, and for persisting user rejections so the same noise doesn't recur on every run | Include ACs for relevance inference from surrounding context, granular per-item approval rather than batch accept/reject, and persistence of user decisions across runs with a mechanism to distinguish "previously declined" from "genuinely new template content" | specs/bug-fix-migrate-project-irrelevant-sections/ |
| When specifying features that create transient local state files in a user-controlled workspace, specs describe the artifact's functional role but omit ACs ensuring it is excluded from version control so downstream generic operations (like bulk stage-and-commit) don't inadvertently publish it | Include ACs that the tool ensures its artifact paths are excluded from version control at startup, handles pre-existing exclusion configuration idempotently, and creates the exclusion mechanism when absent | specs/bug-fix-sdlc-runner-auto-mode-gitignore/ |
| When specifying features that select an item from a collection fetched via an external API, specs assume the happy path (one obvious target) and omit ACs for the data-fetching strategy — what metadata must be retrieved and how filtering occurs before selection, versus probing sequentially | Specify what metadata is fetched upfront to enable filtering, how empty/invalid candidates are excluded before user-facing selection, auto-selection behavior when exactly one viable candidate remains, and fallback behavior when zero candidates match the filter | specs/bug-fix-start-issue-milestone-selection/ |
| When specifying features that read from optional upstream artifacts produced by earlier workflow steps, specs assume the full workflow was followed and omit ACs for the "artifact absent" branch — causing the feature to hard-fail when a user legitimately skipped upstream steps | Include ACs specifying detection of artifact absence before reading, a concrete fallback data source when artifacts are missing, output adaptations (omitted sections, relabeled sources, warning notices) when fallback is used, and preservation of happy-path behavior when artifacts are present | specs/bug-handle-missing-spec-files-in-open-pr/ |
| When specifying orchestrators that evaluate success/failure of subprocesses, specs tend to define success only via structured signals (exit codes, JSON fields) and omit ACs covering the case where a subprocess exits successfully but emits textual failure markers on stdout/stderr, allowing silent failures to cascade | Define the full failure-detection surface in ACs — structured signals AND textual output patterns; require the failure-pattern catalog be explicit, maintained in one place, and routed through the same retry/escalation logic as structured-signal failures; add regression ACs that normal successes are not misclassified | specs/bug-text-pattern-soft-failure-detection-sdlc-runner/ |
| When specifying features that mutate shared workspace state (branches, working directories, databases), specs focus on the intended transformation but omit ACs covering preconditions about ambient/inherited state, letting pre-existing uncommitted or polluted state silently leak into the new artifact | Specify the exact cleanliness/readiness predicate for the source state, the abort behavior and diagnostic content when the predicate fails, the escalation path under non-interactive mode, and a positive regression AC that a clean source proceeds without friction | specs/bug-validate-git-cleanliness-before-branching-in-start-issue/ |

## Undertested Boundaries

Defects caused by boundary conditions the original feature spec addressed
insufficiently.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When specifying features that delegate work to subordinates (subprocesses, external tools, or LLM-based agents executing prose instructions), specs trust the delegate's self-reported success signal (exit code, structured status field, "I completed the instructions") — subordinates can exit cleanly despite soft failures, partial completion, or silent omission of required steps, and specs without postcondition verification promote these to pipeline successes | For every delegation step, pair the delegate's self-report with independent postcondition verification: (a) inspect all structured output fields the delegate can use to report incomplete work (max-turns, quota exhaustion, permission denials, partial completion), (b) confirm the expected side-effect artifact/state change exists (files written, branches pushed, commits produced, version numbers updated), and (c) retry with escalating prescriptiveness when the postcondition fails | specs/bug-detect-soft-failures-runner-tests/, specs/bug-fix-silent-commitpush-failure/, specs/bug-fix-inconsistent-version-bumping/ |
| When specifying features that create transient state during a run, specs cover the happy-path lifecycle (create on start, remove on normal completion) and omit ACs for every non-happy exit route: signal-driven shutdown, escalation/abort, unhandled exception, and early-exit modes — state leaks across sessions because only the happy path was specified | Enumerate every termination path (graceful, signal, error, escalation, single-step, crash) as separate ACs requiring cleanup; require cleanup logic to be centralized so new exit paths inherit it; include a regression AC that state is still present during active execution | specs/bug-fix-auto-mode-cleanup-on-exit/ |
| When specifying orchestration features where one phase monitors an asynchronous signal, specs describe "observe and report" behavior without ACs that commit the phase to a remediation outcome — the phase exits "successfully" by merely reporting the adverse state, and the pipeline thrashes between this phase and the next precondition gate | For monitoring/gating phases, specify that the phase's exit code is tied to the desired terminal state (not to reporting completion), allocate sufficient resource budgets (turns, timeouts) for remediation rather than observation alone, and name explicit escalation criteria when remediation would require deviating from spec | specs/bug-fix-monitorci-step8/ |
| When specifying features that loop over multiple work units with shared runtime state, specs say "reset state between iterations" at a high level but omit ACs about the data source used to derive identifiers for each new iteration — deriving identifiers from free-form prose output (which carries context from prior iterations) rather than deterministic canonical sources produces silent cross-iteration contamination | Specify that identifiers for each iteration are extracted from deterministic ground-truth sources (filesystem, branch name, database row) rather than parsed from LLM/narrative output; require the working environment be affirmatively cleaned between iterations (not just "state reset"); add post-extraction validation that the derived identifier matches the canonical source | specs/bug-fix-sdlc-runner-cross-cycle-state-contamination/ |
| When specifying features that poll an external system for a status signal, specs enumerate "pass" and "fail" paths but omit the "signal absent / not configured" path — tools returning a non-zero exit for "no data" are conflated with "failed data," producing infinite retry loops in environments where the capability simply isn't deployed | Distinguish three states in ACs — present-and-passing, present-and-failing, and absent/not-configured — and specify behavior for the absent case explicitly (typically "treat as pass-through" or "skip gate with warning") rather than letting it fall through to the failure handler | specs/bug-fix-sdlc-runner-infinite-retry-no-ci/ |
| When specifying a system with multiple components that share a working directory where some components write runtime state/artifact files, specs fail to define which files are "noise" that sibling components must tolerate, leading components to treat their peers' artifacts as user-authored changes | Enumerate the known runtime artifacts in ACs, require filtering those artifacts from dirty/diff checks, preserve rejection of real user changes, and require untracking any runtime artifact that was historically committed | specs/bug-fix-start-issue-dirty-tree-runner-artifacts/ |
| When specifying features that populate a traceability/cross-reference field, specs describe the field's existence and format but not the active discovery procedure required to populate it — leaving the field technically supported but practically always empty because no AC mandates a search step | Specify the concrete discovery procedure (inputs, search surfaces, matching strategy), what populates the field when a match is found, and the regression case where no match exists and the field is explicitly set to a null value | specs/bug-fix-write-spec-defect-related-spec-search/ |

## Domain-Specific Gaps

Defects caused by domain knowledge the original feature spec failed to capture.

| Learning | Recommendation | Evidence (defect specs) |
|----------|---------------|------------------------|
| When a feature composes a prompt or instruction that must be interpreted by a downstream parser (shell, DSL interpreter, LLM-based invocation), specs describe the semantic content but omit ordering/positional constraints that govern whether the target will recognize the invocation at all — prefix vs suffix placement of command tokens can be the difference between "executed" and "treated as prose" | Pin the exact structural position of invocation tokens (first line, leading character) and separately specify where instructional modifiers/constraints are appended; include an AC for environments where the target has strict recognition rules (e.g., disabled implicit inference) and must see the invocation in a canonical form | specs/bug-exercise-template-dry-run-prefix/ |
| When specifying features that traverse cross-document reference links, specs assume well-formed graphs and omit ACs for malformed, circular, or wrong-type references — discovery/traversal ACs lack validation, chain resolution, and type-filtering requirements, surfacing as silent incorrect behavior in production | Include ACs for reference target validation (exists and is the expected type), chain resolution with cycle detection to reach a canonical root, type-filtering when populating reference fields so incorrect categories aren't selected, and explicit behavior on broken or orphan chains | specs/bug-fix-run-retro-defect-spec-discovery/ |
| When specifying features that synthesize insights from historical artifacts, specs define the output schema and source inputs but omit ACs controlling the abstraction level and aggregation behavior — without "generalize, don't transcribe" and "merge recurring patterns" requirements, the output becomes a catalog of source-specific notes rather than transferable principles | For synthesis features that produce guidance from historical data, specify the required abstraction level (domain-agnostic vs. domain-specific), cross-source pattern aggregation with evidence traceability, and forward-looking framing ("apply to future cases") rather than backward-looking framing | specs/bug-fix-run-retro-learnings-transferability/ |
| When specifying features that spawn subprocesses which can themselves spawn child processes (browsers, shells, multi-process tools), specs omit ACs governing the process-tree lifecycle — how entire descendant trees are tracked, terminated, and scoped so cleanup doesn't affect unrelated host processes | Specify that cleanup kills the full descendant tree (not just the direct child), is scoped to processes the feature itself spawned (never matches patterns that could hit unrelated user processes), handles orphaned/reparented-to-init descendants, and always emits observable log output whether it found matches or not | specs/bug-fix-sdlc-runner-process-tree-cleanup/ |
| When specifying features that classify tool/subprocess output into success/failure categories using pattern matching, specs omit the contextual dimensions of classification (e.g., the target path of a denied operation, not just the operation name), causing benign expected failures to be misclassified as hard errors | Specify which structured fields contribute to the success/failure verdict, benign-exception carve-outs (by path, scope, or category), that an expected/sandboxed operation against a known-benign target is NOT classified as failure, and that mixed batches still escalate when a non-benign signal is present | specs/bug-fix-verify-step-false-escalation-on-temp-dir-denials/ |
| When specifying features that write to filesystem locations owned or protected by the host environment/platform, specs rarely capture constraints the host imposes on those paths — features work today but break when the platform tightens directory protections, with no defined migration path | Explicitly distinguish user-content paths from host/platform-reserved paths; require user content live outside directories the host may restrict; define a supported upgrade/migration procedure when directory conventions change; include a hard-gate that refuses operation on legacy layouts with an instructive remediation message | specs/bug-relocate-specs-and-steering/ |
| When specifying features that perform migrations or transformations over heterogeneous legacy artifacts, specs describe the end-state naming/structure convention but omit ACs for concrete detection-and-rename mechanics on pre-existing instances, and conflate reversible rename operations with irreversible merge operations under a single "destructive" label — causing automated modes to skip safe work | Specify the exact detection predicate for each legacy variant, the tool/command used to perform the transformation (history-preserving move vs. delete+create), downstream cross-reference updates needed after the primary change, and a destructiveness/reversibility classification so non-interactive modes can safely auto-apply low-risk transformations while gating high-risk ones | specs/bug-spec-directory-rename-migrate-project/ |
